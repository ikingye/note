    Python #

    • 对 CPU 密集型代码（比如循环计算），多进程效率更高
    • 对 IO 密集型代码（比如文件操作，网络爬虫），多线程效率更高
      • CPU 非瓶颈，而且没有进程切换那么大消耗

    Python 多进程 #

    进程间通信 #
    1. 管道 / 匿名管道 (pipe)
      • 管道的实质是一个内核缓冲区
    2. 有名管道 (FIFO)
      • 有名管道的名字存在于文件系统中，内容存放在内存中
    3. 信号
    4. 信号量
      • 信号量是一个计数器，用于多进程对共享数据的访问
    5. 套接字
    6. 共享内存
    7. 消息队列
      • 消息队列存放在内核中，只有在内核重启 (即，操作系统重启) 或者显示地删除一个消息队列时，该消息队列才会被真正的删除
    multiprocess #
    Process # 
    from multiprocessing import Process, Queue

# 使用队列进行数据共享和通信
# 父进程创建 Queue，并传给各个子进程
q = Queue()
p1 = Process(target=long_time_task, args=(q,))
p1.start()
# 等待进程结束
p1.join()
# 若 long_time_task 是死循环，则无法等待进程结束，只能强行终止
# p1.terminate()
    Pool # 
    from multiprocessing import Pool, cpu_count

cpu_cnt = cpu_count()
p = Pool(cpu_cnt)

for i in range(100):
	p.apply_async(long_time_task, args=(i,))

# 调用 join () 之前必须先调用 close () 或 terminate () 方法，让其不再接受新的 Process 了
p.close()
p.join()

    Python 多线程 #

    • 由于 GIL 的存在，Python 的多线程并非真正的多线程
    • Python 一般使用多进程
    threading #
    threading.Lock # 
    import threading

# 对共享变量进行锁操作
lock = threading.Lock()
# lock.acquire()
# lock.release()

t1 = threading.Thread(target=long_time_task, args=(lock,))
# 主线程结束时不再执行子线程
# t.setDaemon(True)
t1.start()
t1.join()
    queue.Queue # 
    from queue import Queue

# 如果不设置长度，默认为无限长
queue = Queue(5)
# 如果队列已满，此方法将阻塞至队列有空间可用为止
queue.put(1)
queue.get()

# queue 同时还自带 emtpy (), full () 等方法来判断一个队列是否为空或已满，
# 但是这些方法并不可靠，因为多线程和多进程，在返回结果和使用结果之间，队列中可能添加 / 删除了成员。

    GIL (Global Interpreter Lock) #

    • 全局解释器锁是计算机程序设计语言解释器用于同步线程的一种机制，它使得任何时刻仅有一个线程在执行。
      • 即便在  多核心处理器上，使用 GIL 的解释器也只允许同一时间执行一个线程。
    • GIL 并不是 Python 的特性，它是在实现 Python 解析器 (CPython) 时所引入的一个概念
    • CPython 的线程是操作系统的原生线程。
      • 在  Linux  上为 pthread，在  Windows  上为 Win thread，完全由操作系统调度线程的执行。
    • 一个 Python 解释器进程内有一个主线程，以及多个用户程序的执行线程。
      • 即便使用  多核心 CPU  平台，由于 GIL 的存在，也将禁止多线程的并行执行。
    • GIL 的存在是为了方便使用 C 语言编写 CPython 解释器的编写者
      • 而顶层使用 Python 时依旧要考虑线程安全
    • 常见的使用 GIL 的解释器有：
      • CPython
      • Ruby MRI
    为什么会有 GIL #
    • GIL 的存在更多的是历史原因
    • 默认 python 内部对象是 thread-safe 的
    如何绕过 GIL #
    • 多进程
    • C/C++ 来实现这部分代码，并生成对应的 so 或 dll 文件，再通过 Python 的 ctypes 将其调用起来
      • 如 Numpy、Pandas 等
    为什么不去掉 GIL #

    考虑把 GIL 移除之前，不如考虑一下如何给 CPython 加上 JIT

    • 为什么 Python 的 GIL 问题一直让人诟病，Python 社区却不解决？
